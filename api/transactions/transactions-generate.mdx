## Overview

The `transactions_generate` endpoint creates transaction records for an order, updates ticket statuses to sold, and associates tickets with user accounts. This function is typically called automatically by the `order_process` function.

<Warning>
  This endpoint is designed to be called by the order processing system. Manual calls should be avoided unless you're implementing custom order workflows.
</Warning>

## Processing Flow

1. **Create Transaction Records** - Stores transactions in `orders_transactions` collection
2. **Create Custody Account Transactions** - Links transactions to custody accounts
3. **Update Ticket Status** - Marks tickets as sold and updates ledger
4. **Link to User Account** - Associates tickets with user's account

## Transaction Record Structure

Each transaction is stored with the following structure:

```javascript
{
  id: "trans_123",
  amount: 150.00,
  client_id: "client_789",
  client_name: "Event Productions LLC",
  custody_account: {
    id: "acc_001",
    name: "Main Account",
    account_number: "1234567890"
  },
  date: {
    created: Timestamp,
    updated: Timestamp
  },
  event_id: "evt_123",
  event_name: "Summer Concert",
  office_id: "off_001",
  office_name: "Main Office",
  order_id: "order_456",
  payment_data: {
    email: "john@example.com",
    name: "John Doe",
    phone: "+1234567890"
  },
  payment_id: "pay_card_001",
  payment_name: "Credit Card",
  status: true,
  amount_currency: "USD",
  amount_exchange: 150.00,
  amount_exchange_rate: 1.0
}
```

## Ticket Ledger Updates

When tickets are marked as sold, their ledger is updated with a new entry:

```javascript
{
  date: Timestamp,
  action: "sold",
  metadata: "{}"
}
```

The ledger maintains a complete history of ticket state changes:
- `created` - Ticket was created
- `reserved` - Ticket was reserved
- `sold` - Ticket was sold (added by this function)
- `cancelled` - Ticket sale was cancelled
- `used` - Ticket was scanned/used for entry

## Custody Account Transactions

Each transaction is also stored in the custody account's transaction subcollection:

```
custody_accounts/{account_id}/transactions/{transaction_id}
```

With additional fields:
- `amount_type`: "payment"
- `order_transaction_id`: Reference to the main transaction document

## User Ticket Association

Tickets are copied to the user's account at:

```
u_users/{user_uid}/tickets/{ticket_id}
```

This allows users to:
- View their purchased tickets
- Access ticket details and QR codes
- Manage ticket transfers

<Warning>
  If a user with the customer_email doesn't exist, the ticket linking will fail silently. Ensure users are created via the order_user endpoint before generating transactions.
</Warning>

## Database Operations

The function performs writes to:

1. **Firestore Collections:**
   - `orders_transactions` - Main transaction records
   - `custody_accounts/{id}/transactions` - Account-specific transactions
   - `events/{id}/tickets` - Ticket status updates
   - `u_users/{uid}/tickets` - User ticket copies

2. **PostgreSQL Tables:**
   - `orders_transactions` - Transaction records mirror
   - `tickets` - Ticket status and ledger updates

## Error Handling

<Warning>
  This endpoint does not return detailed error messages. Monitor Firestore write operations and check that all expected documents are created.
</Warning>

Common issues:
- Missing custody account references
- Invalid ticket IDs
- Non-existent user accounts
- Database write permission errors
